# Тестирование и оценка ML (урожайность и полив)

## 1. Что именно оцениваем

| Компонент | Тип | Метрики / критерии |
|-----------|-----|-------------------|
| Прогноз урожайности | `RandomForestRegressor` + `StandardScaler` | MAE, RMSE, R², доля прогнозов в ±15% от целевого `yield` (т/га) |
| Сценарии погоды | Те же признаки, фиксированные культура/почва | Монотонность: «засуха» и «жара» дают меньший урожай, чем «оптимум» |
| Рекомендации полива | Правила `IrrigationModel` | Ожидаемый приоритет при заданной влажности; объём воды не растёт при росте влажности |
| Скорость | Локальный `predict` и опционально HTTP | p50/p95 времени одного прогноза |

**Важно:** обучающие данные по умолчанию **синтетические** (генератор совпадает с `YieldPredictionModel._generate_training_data` в `main.py`). Высокие R² на таких данных **не эквивалентны** качеству на реальных агрономических рядах. Для «исторических данных» в бою нужен датасет с измеренной урожайностью по полям и те же признаки (погода, почва, культура).

## 2. Быстрый офлайн-прогон

Из каталога `backend/analytics-service` (нужны `numpy`, `pandas`, `scikit-learn`):

```bash
python scripts/ml_evaluation_suite.py --all
```

Отдельные блоки:

```bash
python scripts/ml_evaluation_suite.py --accuracy --samples 8000
python scripts/ml_evaluation_suite.py --scenarios
python scripts/ml_evaluation_suite.py --irrigation
python scripts/ml_evaluation_suite.py --latency
```

Код возврата `0` — все встроенные проверки прошли; `1` — есть FAIL.

## 3. Проверка через работающий API

1. Поднять стек (`docker compose up`) и дождаться обучения модели в analytics (логи: `ML model trained`).
2. Метрики как в UI «Метрики модели»:

   `GET http://localhost:8080/api/analytics/model/metrics`  
   (через gateway; нужен JWT, как для остальных защищённых маршрутов.)

3. Задержка HTTP-прогноза:

```bash
python scripts/ml_evaluation_suite.py --api-latency \
  --base-url http://localhost:8080/api/analytics \
  --bearer "ВАШ_ACCESS_TOKEN" \
  --api-n 40
```

Без валидного `--bearer` прогноз может отвечать `401`.

## 4. Исторические данные (рекомендуемый процесс)

1. **Сбор:** таблица полей × сезонов: фактическая урожайность (т/га), культура, тип почвы, агрегированные погодные признаки за сезон (или помесячно), полив.
2. **Разбиение:** по времени или по полям (не смешивать один и тот же сезон в train и test).
3. **Обучение:** заменить или дополнить `_generate_training_data` загрузкой CSV/БД с тем же набором признаков, что в `FEATURES` в `main.py`.
4. **Отчёт:** те же метрики + разбивка по культурам и регионам; калибровочные кривые при необходимости.

## 5. Релевантность рекомендаций полива

- Юнит-тесты совпадают с логикой `IrrigationModel.recommend` в `main.py` (пороги дефицита влаги, критический случай при влаге &lt;35%).
- Дополнительно вручную: согласовать с агрономом тексты `reason` и диапазоны `CROP_OPTIMAL_MOISTURE`.

## 6. Нагрузочное тестирование (опционально)

- Инструменты: `k6`, `Locust` или `ab` на `POST /api/analytics/forecast/yield` с телом из примеров фронта.
- Фиксировать RPS, p95 латентности, долю 5xx; мониторить CPU/RAM контейнера `analytics-service`.
